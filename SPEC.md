# 需求规格书

> 一切讨论以本文为准。本文只说「要什么」，不说「怎么实现」。

## 用户场景

典型的 ADHD + 阿斯伯格用户。
- 启动困难：知道要做什么，但难以开始
- 需要外部推动：不是「被提醒一下」，是「被追问到动起来」
- 工具不能增加负担：如果维护工具比用它省的时间还多，就是失败

## 使用方式

```
微信发一句话 → AI 拆解 + 安排时间 → 出现在我的日历上 → 到点催我开工 → 没开工就追问 → 走神了提醒
```

用户只跟微信说话，只看日历。中间所有处理对用户透明。

## 需求清单

### R1: 远程任务录入

输入：微信自然语言
输出：日历事件（带时间 + 持续时长）

```
"下周五之前交稿，今天练一小时人体，下午修两个bug"
    → 10:00-11:00 人体练习
    → 14:00-16:00 修登录bug
    → 16:00-17:00 修样式bug
```

不接受：要用户自己拖拽、填时间、分标签。

### R2: AI 拆解

大任务 → 小步骤。每一步都有独立的时间段。

```
"登录功能重做"
    → 阶段1: 分析现有代码（今天 2h）
    → 阶段2: 写新逻辑（明天 3h）
    → 阶段3: 测试（后天 1h）
```

不接受：一个未拆解的标题丢在列表里。

### R3: 到点提醒

日历时间到了 → 弹出提醒 → 催促开工。

不只是系统通知。如果用户没反应，要追问。

### R4: 拖延检测

日历上安排了任务但到了时间用户没开始 → 追问"开始了吗？"

追问可以递增：第一次温和 → 第二次直接 → 第三次换方式。

### R5: 走神提醒

正在工作时间，检测到在刷娱乐内容超时 → 提醒。

不需要阻止，只需要提醒。用户可以选择继续休息。

### R6: 工时记录

自动记录每天在各分类上的时间：
- 绘画（CSP、Procreate）
- 编程（VS Code、终端）
- 摸鱼（娱乐网站、游戏）

不需要用户手动按计时器。

### R7: 跨设备同步

- Windows（主力工作机）上能看日历和任务
- iPad 上能看
- 数据一致，不需要手动同步

### R8: 每日摘要

每天晚上或早上：今天/明天的计划摘要，推送到微信。

### R9: 成就反馈

- 连续完成天数
- 本周累计工时
- 鼓励性质的正面反馈

对 ADHD 用户很重要：需要看到进展，否则容易放弃。

## 不需要的东西

- 精细项目管理（标签体系、优先级矩阵、自定义视图）
- 手动排期（拖拽、填表单）
- 团队协作
- 复杂的统计报表
- 番茄钟
- 多层级项目结构

## 工具原则

- 优先现有开源工具，不从头开发
- 脚本/插件/API 缝合适配，不修改工具源码
- 4GB 服务器能跑
- 微信是唯一输入入口
- iOS/Android/Windows 都能查看结果
